## 5. Database

Source of truth: `supabase/migrations/*.sql`.

### Tables (`public` schema, 25 total)

`addresses`, `analytics_events`, `audit_log`, `banners`, `categories`, `contact_messages`, `coupon_redemptions`, `coupons`, `email_log`, `notifications`, `order_items`, `orders`, `payments`, `product_reviews`, `product_variants`, `products`, `profiles`, `return_requests`, `reviews`, `site_content`, `support_ticket_attachments`, `support_ticket_messages`, `support_tickets`, `user_roles`, `wishlist`.

All tables have RLS enabled with explicit `GRANT`s for `authenticated` / `service_role` (and `anon` only on publicly-readable tables such as `products`, `categories`, `banners`, `site_content`).

### Enums

- `app_role` — `'user' | 'seller' | 'admin'`
- Order status (text-constrained, not Postgres enum): `pending`, `confirmed`, `packed`, `shipped`, `out_for_delivery`, `delivered`, `cancelled`, `return_requested`, `returned`, `refunded`
- Refund status: `none`, `pending`, `approved`, `rejected`, `refunded`
- Payment status: `pending`, `paid`, `failed`, `refunded`
- Return reason: `damaged`, `wrong_item`, `not_as_described`, `defective`, `no_longer_needed`, `other`

### Triggers

- `handle_new_user` on `auth.users` → seeds `profiles` row + assigns default `user` role in `user_roles`.
- `trg_*_updated_at` BEFORE UPDATE on every table that has `updated_at` (uses `public.update_updated_at_column()`).

### Database functions (custom; pg_trgm extension functions omitted)

- `public.has_role(_user_id uuid, _role app_role) → boolean` — SECURITY DEFINER role check (used by every RLS policy needing role gating).
- `public.is_order_owner(_user_id, _order_id) → boolean` — SECURITY DEFINER.
- `public.is_order_seller(_user_id, _order_id) → boolean` — SECURITY DEFINER, checks `order_items → products.seller_id`.
- `public.handle_new_user()` — trigger function (see above).
- `public.update_updated_at_column()` — trigger function for `updated_at`.
- `public.decrement_stock(p_product_id uuid, p_qty int)` — SECURITY DEFINER, row-locking stock decrement used at checkout.
- `public.increment_stock(p_product_id uuid, p_qty int)` — SECURITY DEFINER, used on cancel/return.
- Plus `pg_trgm` extension functions for fuzzy product search.

### Storage buckets (all private)

| Bucket | Purpose | Access pattern |
|---|---|---|
| `product-images` | Seller product photos | Sellers write to own folder; signed URLs for display |
| `return-photos` | Buyer return-request photos | RLS scoped to `userId` folder; admin/seller read via signed URL |
| `support-attachments` | Support ticket attachments | Ticket participants + admin via signed URL |

### RLS policy summary

- **user_roles**: read self or admin; only admins (via service role / privileged server fns) mutate.
- **profiles**: read self / public-displayable fields; update self.
- **products / categories / banners / site_content**: public read; admin (+ seller for own products) write.
- **orders / order_items**: owner read; seller reads orders containing their products (`is_order_seller`); admin reads all. Status transitions guarded server-side in `src/lib/orders.functions.ts`.
- **return_requests**: insert by owner; read by owner / order seller / admin; update by admin.
- **payments**: owner + admin read; service role writes (webhook).
- **notifications**: read/update own.
- **support_tickets / messages / attachments**: ticket creator + assigned seller + admin.
- **audit_log**: admin read only; service role / privileged server fns write.
- **wishlist, addresses, coupon_redemptions**: owner-scoped.

## 8. Build Output

`bun run build` (Vite + Nitro) outputs to:

- **`.output/`** — Nitro server bundle. **This is the deployable folder.**
  - `.output/public/` — static client assets (JS, CSS, images)
  - `.output/server/` — SSR server entry (`index.mjs`) for the configured Nitro preset
- `dist/` — only used if a pure SPA build is configured (not the case here)

Default Nitro preset is **Cloudflare Workers**. The Lovable platform deploys `.output/` automatically. For other targets, set `NITRO_PRESET` (e.g. `node-server`, `vercel`, `netlify`) before building.

## 9. Runtime

This is a **server-rendered (SSR) application**, not a static SPA.

- **Framework**: TanStack Start v1 on Vite 7, server entry bundled by Nitro.
- **Server**: Cloudflare Workers (default) — there is no Express, Koa, or long-running Node process. The handler is a single `fetch(request)` function (`src/server.ts` wraps the TanStack Start entry to render a friendly 500 page on catastrophic SSR failure).
- **Client**: React 19 hydrates the SSR HTML. TanStack Router takes over navigation client-side.
- **Server functions**: `createServerFn` modules (`src/lib/*.functions.ts`) are exposed as POST RPC endpoints; the client middleware in `src/start.ts` attaches the Supabase bearer token automatically.
- **Server routes**: `src/routes/api/public/razorpay-webhook.ts` is a raw HTTP endpoint (no auth — verifies HMAC signature internally).
- **Auth-gated SSR**: routes under `src/routes/_authenticated/` use `ssr: false` (session lives in `localStorage`, unreachable from the server). All other routes SSR by default.

Not: static SPA, Next.js, Express, plain Vite dev-server in production.

## 10. Production Checklist

### Before first deploy
- [ ] Verify all server-side secrets are set in the hosting environment (see §3): `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`, `RAZORPAY_WEBHOOK_SECRET`.
- [ ] Confirm `.env` (committed) contains only the three `VITE_SUPABASE_*` publishable values — never a service-role key.
- [ ] Run all migrations against the production Supabase project (`supabase/migrations/` in chronological order).
- [ ] Confirm storage buckets exist and are **private**: `product-images`, `return-photos`, `support-attachments`.
- [ ] Confirm at least one user is granted the `admin` role in `user_roles` (cannot self-promote through the UI).
- [ ] In Supabase Auth → Providers, enable **Email** and **Google**. Google requires Client ID / Client Secret configured at the project level.
- [ ] Add the production domain to Supabase Auth → URL Configuration → Site URL and Redirect URLs (including `https://<domain>/` and `https://<domain>/auth`).
- [ ] In Razorpay dashboard, configure the webhook endpoint to `https://<domain>/api/public/razorpay-webhook` with the same `RAZORPAY_WEBHOOK_SECRET`. Subscribe to `payment.captured`, `payment.failed`, `refund.processed` events.
- [ ] Disable Supabase "auto-confirm email" only if you intend to require email verification; otherwise leave on for smoother onboarding.
- [ ] Disable anonymous sign-ups in Supabase Auth (not used by this app).

### Operational
- [ ] HTTPS enforced on the production domain (required for Auth cookies and OAuth).
- [ ] Set sensible Razorpay rate limits / fraud rules.
- [ ] Monitor Supabase usage (DB rows, storage GB, egress, MAU) against plan limits.
- [ ] Configure database backups in Supabase (PITR if available on your plan).
- [ ] Test the full marketplace flow end-to-end against production: register → become seller → admin approves → list product → buy → pay → seller progresses order → admin handles return/refund.
- [ ] Verify the audit log is being written on every status transition (`audit_log` table).

### Security hardening
- [ ] Run `supabase--linter` and resolve all WARN/ERROR findings.
- [ ] Confirm RLS is enabled on every table in `public` (no exceptions).
- [ ] Rotate `LOVABLE_API_KEY` and any leaked keys via the Lovable rotate tool before going live.
- [ ] Ensure no service-role key appears in client bundles (search built `.output/public/` for `sb_secret`).
- [ ] Confirm webhook signature verification is active (don't disable for "testing" in prod).

### Post-deploy verification
- [ ] Smoke-test sign-in (email + Google), checkout (test Razorpay key), order detail, admin returns screen.
- [ ] Confirm SSR works (view-source on the home page should contain rendered product markup).
- [ ] Confirm the `/api/public/razorpay-webhook` endpoint rejects unsigned POSTs with 401.
- [ ] Trigger a return request and verify only the order owner sees the form (admin/seller see read-only details).
